2D Game Engine Documentation

This engine handles 2D image loading and rendering, simple collision detection, map loading and drawing and UI loading and drawing.

3rd Party Libraries

GLFW 3.0, which has been modified slightly to add some win32 window handle modification.

GLEW_MX, used to handle OpenGL extensions.

FreeImage, handles the loading of images into the framework.

TinyXML2, handles XML parsing and loading.

Python, for scripting AI behaviours.

Known issues

  • Collision with some map objects doesn't get called.

    • This is because of the quadrant system used to minimise collision calls to only nearby objects

  • No render order for objects

    • This hasn't been implemented yet, z-ordering still needs to be implemented

  • Pathfinding occasionally cuts corners or backtracks

    • This is because of a poor implementation of A*, a more efficient approach is needed.

  • Wandering AI behaviour just dances

    • This is because it really just likes dancing, also because the wandering algorithm needs to be fixed so that it only picks directions in front of itself.

AI Behaviour documentation

Behaviours in place in the engine

Flee

Moves an entity away from a set target when they are within a certain range. There is very basic obstacle avoidance assisting this. Possible improvements could include

improvements to how far away it will move as well as moving back towards its starting position when it is 'safe' to do so.

This is implemented by getting a vector of the distance between the two positions, then negating and normalising it to recieve a velocity for the entity.

Follow

Moves an entity towards a target position or object. Very basic obstacle avoidance assists the path finding. Could be imporved with A* path finding to always

travel along the most efficient path to its target.

This is implemented by getting a vector of the distance between the two positions, then normalising it to recieve a velocity for the entity.

Wander

This behaviour is meant to randomly steer an entity around a map avoiding obstacles. Instead the entity moves slightly and appears to be having a seizure.

This can be improved by it working and a better positioning system to determine the next step an entity takes.

At the moment this is just calculated by a switch statement that generates a random number which is translated into a direction to move. Because this happens every frame it

gives the impression of the entity having a fit. The correct implementation involves only picking a direction in front of the entity to move so that there is no random

jumping backwards or to the side.

Behaviours yet to be implemented

Arrival

Slows an entity down as it arrives at a destination. This works by determining the distance to the target and factoring in a deceleration value to the total speed the

entity needs to be travelling.

Pseudo-code: Velocity = DirectionToTarget * ( SpeedOfEntity / DistanceToTarget )

Alignment

Aligns the direction of an entities velocity with its neighbours. This is done by averaging any related entities direction of movement and then subtracting

the main entities direciton to get he force in which it should be travelling. This can be used with separation to maintain a distance between neighbours to

create interesting group movement.

Pseudo-code: AverageOfNeighboursDirection - CurrentDirection = NewDirection

Flocking

Moves a selection of entities towards each other or within a distance of each other, often used with alignment and wandering behaviours to create emergent behaviour.
